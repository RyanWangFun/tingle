# on-trend-radar

## 概述与核心原则

本 skill 只做一件事：**读今日热点 × 某项目的选题上下文，逐个热点判断「有没有一座真实的桥」能接到该项目的人群痛点上，接得住产出一条具体选题、接不住就丢，写入选题清单交人复核。**

**核心原则 —— 价值在判断，不在拼数量。** 逐个热点问「这条热点和我的人群痛点之间，有没有一座站得住的真桥」；有桥才出一条选题，没桥就丢，绝不把每个热点硬凑成选题。一份三条真选题的清单，胜过十条牵强填数的清单。skill 的产物是「待复核选题」，采纳与否是使用者的人工闸门，本 skill 只写、不替人拍板。

**适用环节**：本 skill 对应内容生产流程里「选题」这一可复用环节——从人群痛点 + 当日热点产出选题清单，跨项目复现（每天可跑、每个项目可换），不是一次性事务。

**这一环节的边界（记牢，越一步即错）**：本 skill 止于「选题」——一句话把热点接到一个具体人群痛点上的主题。**它绝不产出「内容方向 / 内容形态 / 切角」**，这三件是下游三个独立环节。切角是本 skill 的**输入**（品牌惯用切入角度，用来判断「这条热点顺哪个角度接得住」），**不是输出**——读它，不产它。

## 前置依赖

本 skill 依赖 lark 工具链（飞书读写）完成「读热点榜 + 读项目选题上下文」与「写选题清单」。**运行时第一步**须先自检 `command -v lark-cli`：

- **自检通过**（命令存在）：正常进入下述四步引擎。
- **自检不过**（命令缺失）：立即停止，打印安装指引，不抛出难懂的报错——
  1. 安装 lark-cli：`npx skills add larksuite/cli -g`；
  2. 执行 `lark-cli auth login` 完成飞书授权；
  3. 确认账号对本次要读写的相关飞书表（当日热点榜 / 项目选题上下文文档 / 选题清单表）具备访问权限。

三步齐备后重新运行本 skill。

## 何时用 / 何时不用

**何时用**
- 每天要把当日热榜转成某项目的 on-trend 选题清单，替代人工「盯着热搜憋选题」。
- 已有一份该项目的「项目选题上下文」文档（含人群痛点、认知主线意图集、惯用切角等 8 个字段）且挂了 [品类 / 品牌] 标签。
- 需要一个项目无关、换个项目标识就零改复用的通用选题引擎。

**何时不用（When NOT）**
- **要定内容方向 / 内容形态 / 切角** —— 这三件是下游独立环节，本 skill 只交「选题」，越到任一件即错。
- **要出成稿或做合规终审** —— 选题 ≠ 成稿；权威合规审稿是下游独立环节，本 skill 只做合规轻筛提示、不做权威门。
- **该项目还没有「项目选题上下文」文档** —— 本 skill 不自己蒸馏、不现读原始立项材料。此时**停并报「缺项目选题上下文，需先蒸馏」**，不猜路径、不硬跑。
- **要采集或修改热点榜** —— 本 skill 对热点榜只读，不采、不改。

## 方法：四步引擎

### 入参与接口

**入参**（本 skill 的调用契约）= 项目标识 → 解析到三样：
1. 该项目「项目选题上下文」飞书云文档**链接（或 token）** —— 用 lark-doc 只读；
2. 项目 **[品类 / 品牌] 标签** —— 据此从四层规则叠加加载对应层（见「依赖」）；
3. 选题清单**写入分区**（用项目标识给清单分区）。

**缺链接即停**：未拿到「项目选题上下文」文档链接，停并报 `缺项目选题上下文，需先蒸馏`，不猜路径、不自蒸馏。

### 依赖

**依赖的规则层与项目事实**：运行时按项目 [品类 / 品牌] 标签，从四层规则中**叠加加载两层** ——
- **品类层**：合规红线、品类通用规则（随 [品类] 标签变，**不写死任何具体品类**）；
- **品牌层**：调性红线、品牌禁用表述（随 [品牌] 标签变）。

所需**项目事实字段**（全部来自「项目选题上下文」文档，本 skill 只读、不内嵌）：项目身份、目标人群 + 核心摩擦点、认知主线 · 意图集、品牌惯用切角、转化终点 CTA、惯用内容形态、合规 · 调性红线、源材料 + 蒸馏日期，共 8 字段。字段级判据见 `references/io-contract.md` 第 1 节。

**依赖的工具 / 接口**：
- `lark-doc` —— 只读「项目选题上下文」飞书云文档；
- `lark-base` —— 只读当天热点榜、写入选题清单表（写为对外动作，须先放行，见升级条件）。

### 输入 / 输出 schema

- **input schema =「项目选题上下文」**（8 具名字段，如上）—— 类型与颗粒度判据以 `references/io-contract.md` 第 1 节为准。
- **output schema =「on-trend 选题清单」一行**：5 内容字段（热点原文+来源 / 热点日期 / 热点回溯锚 / **选题** / 对应人群+痛点 / **为哪段意图作证** / 合规红线提示；其中「热点来源束」展开为前 3 个可存字段）+ 3 管理字段（项目标识 / 采纳状态 / 生成日期）= 10 可存字段 —— 字段类型、格式、取值域以 `references/io-contract.md` 第 2 节为准。**输出中没有「内容方向 / 内容形态 / 切角」任何字段。**

### 四步

**Step 1 · 读**
- 用 lark-base 读**当天**热点榜（只读）；
- 用 lark-doc 读该项目「项目选题上下文」文档，取全 8 字段（尤其记住：输②人群卡点、输③意图集、输④惯用切角、输⑦项目红线）；
- 按 [品类 / 品牌] 标签叠加加载品类层 + 品牌层规则。

**Step 2 · 判断（逐个热点问「真实的桥」）** —— 本引擎的重心，判断法见下节「真实的桥」。

**Step 3 · 产一条 / 丢**
- 过了「真实的桥」四探针 + 四质量门 → **产一条选题**（按 output schema 写这一条，字段值须与上下文咬合，见下节示例）；
- 任一探针或质量门不过 → **丢弃这条热点，不硬凑**。宁可今天零选题，不出牵强选题。

**Step 4 · 写选题清单**
- 每条选题按 output schema 写入选题清单表，按项目标识分区；
- 「采纳状态」默认写 `待复核`，**本 skill 绝不改写此字段**（采纳 / 弃用是使用者的人工闸门）；
- 写入前先过**防幻觉回读闭环**（见下）。

### 「真实的桥」判断法（本引擎的质量本体）

**要判的是同一个分野：接得住（有真桥）还是牵强（空中的板）。**

- **接得住** = 热点之所以热的那批人、那根神经，正好就是本项目人群卡在那个痛点上的那批人、那根神经；顺着品牌惯用的某个切入角度，能自然接上，并且接上后能兑现意图集里某一条具体主张。
- **牵强** = 热点和项目之间只靠一个**表层关键词、谐音或「也能顺嘴提一句」**连着；追热点的人和你的人群其实不是一批，或者热点的情绪根本没碰到那个卡点。典型露馅信号：要接上，只能新造一个意图集**之外**的意图（如「蹭流量涨声量」），或只能靠一个共同名词硬拗。

**逐个热点，四道探针全过才算有桥（任一道断，即牵强，丢）：**

1. **同一批人**：让这条热点热起来的是谁？把这批人对照输②的人群层——他们是否**覆盖或包含**本项目某个目标人群层？若追这条热的人和你的人群没有稳定重叠，桥是架在空中的板。
2. **同一根神经**：这条热点**为什么**热——驱动它的受众情绪是什么（共鸣 / 愤怒 / 焦虑 / 好奇 / 认同…）？这股情绪是否**正好触到**输②里那个人群层的具体卡点？只共享一个名词、但情绪与卡点无关，不算过。
3. **意图作证**：如果就这条热点立一条选题，它能不能**自然植入输③意图集里的某一条**具体主张？须能逐字指认是「认知主线X：某原句」。指认不出、或只能新造意图集外的意图 → 断（防孤儿，下游「定内容方向」会把挂不回意图的方向砍掉）。
4. **顺角度接**：读输④品牌惯用切角，是否**至少有一个**惯用切入立场 / 视角，能自然接住这条热点？若必须动用品牌从不用、且明显外接的立场才勉强搭上，桥是拧上去的。（**注意：这一探针只是读切角判「接得住」并塑选题的切入取向，绝不产出任何切角字段。**）

**四质量门（探针过后，仍须四门全过，才算「接得住」，才产选题）：**

| 质量门 | 判据（可判句） |
|---|---|
| 结合真 | 桥站在真实共同点上（同一批人 + 同一根神经，探针 1、2 过），不是关键词巧合。 |
| 可执行 | 下游「定内容方向」一眼就能接着做——选题足够具体，不是「做个测评 / 做个科普」这类泛表述。 |
| on-trend | 选题的时效价值**依赖这条热点当下的势能**——把热点拿掉，这条选题就失去当下性。若同一句话哪天发都行，那是常青填充，不是 on-trend。 |
| 合规调性 | 选题不越 [品类 / 品牌] 标签加载的品类层 + 品牌层红线，叠加输⑦项目补充红线（轻筛提示，非权威门）。**红线一律运行时按标签加载，本 skill 正文不写死任何品类的红线条目。** |

**吃劲概念，落到操作层（不留含糊、不循环定义）：**
- **接得住**：过四探针 + 四质量门，可逐条打勾——不是「感觉能接」，是一张核对表。
- **牵强**：连接只经表层关键词 / 谐音 / 「顺嘴提一句」，而人群或情绪并不真重叠；露馅信号 = 只能靠共同名词硬拗或新造意图集外的意图。
- **意图作证**：这条选题若做出来，能自然放进输③意图集里**某一条原句**，且能逐字指认是哪条；指认不出即断。
- **on-trend**：把热点抽掉，选题就不再有当下性——反证法一试即知。
- **切角切入（输入侧）**：读输④判「顺哪个角度接得住」，用它塑选题的切入取向；**读它、不产它**。

**一正一反、项目无关示例（占位符）：**

正例（有真桥 → 产一条选题）
- 热点：`[热点事件：某公众人物公开谈自己曾长期误解某问题、后来发现另有其因]` · `[来源平台]`；热度来自 `[受众情绪：大量普通人共鸣「原来我也一直误会了」]`。
- 上下文：输② `[人群层A] —— 卡点：以为某问题只能靠某旧方案，不知另有解法`；输③ `认知主线①：某新解法其实存在`；输④ `从 [某身份视角] 切入`。
- 四探针：① 同一批人 ✓（让它热的正是「曾误解某问题」的普通人 = 人群层A）；② 同一根神经 ✓（热点情绪「我一直误会了」正好触到人群层A 的错误认知卡点）；③ 意图作证 ✓（顺势可植入认知主线①）；④ 顺角度 ✓（品牌惯用「从 [某身份视角] 切入」，正好用这个视角接）。
- 四质量门：结合真 ✓ / 可执行 ✓ / on-trend ✓（借当下共鸣势能）/ 合规 ✓（按标签红线核，无越线）。
- → 产：`借 [热点事件]，面向 [人群层A] 立一条选题：[人群层A 那个错误认知所指的问题，其实存在被忽视的另一面]`。为哪段意图作证 = `认知主线①：[逐字引用输入③ 认知主线① 原句]`。

反例（空中的板 → 丢弃，不硬凑）
- 热点：`[热点事件：某明星恋情官宣]` · `[来源平台]`；热度来自 `[受众情绪：吃瓜、八卦好奇]`。
- ① 同一批人 ✗（追这条八卦的人群与人群层A 无稳定重叠）；② 同一根神经 ✗（八卦好奇与人群层A 的认知卡点无关，唯一连接是硬造名词「恋爱也要注意某问题」）；③ 意图作证 ✗（要接上只能新造意图集外的「蹭流量涨声量」= 孤儿）。
- 判定：牵强 → **丢弃**。任一探针断即丢，不为凑数硬立。

### 防幻觉回读闭环

每写一条选题**之前**：
1. 取该热点在热点榜的稳定行键 `record_id`（= 输出「热点回溯锚」）；
2. 按此锚**回读**热点榜对应行，把将要写入的「热点原文 + 来源」与该行**逐字符比对**；
3. **对得上才写**该选题；**对不上即弃**这条选题，绝不凭记忆或概括写热点原文。

### 决策档 · 复核包 · 升级条件

- **决策档 = 复核**。本 skill 产出「待复核选题」清单，由使用者复核采纳；skill 不自动落定、不替人拍板。
- **复核包（交人复核时随附）**：每条选题附上 —— ① 热点原文 + 来源 + 回溯锚（可追溯凭据）；② 选题本身；③ 对应人群 + 痛点；④ 为哪段意图作证（引用的意图集条目）；⑤ 合规红线提示；⑥ 所据「项目选题上下文」文档链接与蒸馏日期（供核新鲜度）；⑦（可选）当日被丢弃热点的简要弃因，便于复核者理解取舍。
- **升级条件（命中任一即停、交人裁决，不自行推进）**：
  1. **合规红线不确定** —— 某选题是否越 [品类 / 品牌] 红线判不准；
  2. **四层规则冲突** —— 加载的品类层与品牌层红线相互打架，无法自洽取舍；
  3. **缺关键项目事实** —— 上下文文档缺失、意图集为空、或人群卡点无法定位；
  4. **输出需对外发送** —— 选题清单要发到项目外部、或写入 / 改动 Base 等对外不可逆动作，先放行再执行。

### 写回位置与可沉淀候选

- **写回位置**：选题清单表（按「项目标识」分区 / 分视图）。
- **可沉淀候选（reusable_candidates）**：output schema 中标「是」的三字段 —— **选题 / 对应人群+痛点 / 为哪段意图作证** —— 经复核采纳后可回流资产库复用；其余字段（热点束、合规红线提示、三管理字段）标「否」，不回流。

### 产出验收判据（DoD）

一条选题**合格**，当且仅当以下全部成立：
1. 通过「真实的桥」四探针 + 四质量门（结合真 / 可执行 / on-trend / 合规调性）；
2. 「为哪段意图作证」逐字落在该项目输入③意图集内，指认得出是哪条（非新造孤儿意图）；
3. 「热点原文 + 来源」按回溯锚回读、与热点榜对应行逐字符一致；
4. 只到「选题」为止 —— **不含切角 / 内容方向 / 内容形态**任何字段；
5. 「采纳状态」= `待复核`（未被 skill 改写）。
> 内容层面的具体对错（哪些表述越红线）交由运行时按标签加载的四层规则承载（见前述「依赖」一节的规则层加载说明）；本判据本身可据以判定某条选题合格与否。

### 边界（越界即错）

- 不把每个热点硬变选题（判断优先于凑数）；不产成稿。
- **不产下游三环节字段**：内容方向 / 内容形态 / 切角。切角是输入、不是输出。
- 不替使用者拍板采纳（只写「待复核」）。
- 不采热点、不改热点榜；不自己蒸馏、不现读原始立项材料；不改四层规则。
- 无项目 · 品类硬编码：项目 / 品类差异一律经项目标识 + [品类 / 品牌] 标签 + 上下文文档注入。

| Step 2 判 | 逐个热点过四探针（同一批人 / 同一根神经 / 意图作证 / 顺角度）+ 四质量门 |
| Step 3 产/丢 | 全过 → 产一条选题；任一断 → 丢，不硬凑 |
| Step 4 写 | 按 output schema 写选题清单，采纳状态默认「待复核」、不改写；写前回读比对 |
| 防幻觉 | 每条选题写前按 record_id 回读热榜行、逐字符一致才写，对不上即弃 |
| 决策档 | 复核（产待复核项，人裁采纳） |
| 必停升级 | 合规不确定 / 四层规则冲突 / 缺关键项目事实 / 需对外发送 |
| 可沉淀 | 选题 / 对应人群痛点 / 为哪段意图作证 三字段采纳后回流 |
| 不产 | 内容方向 / 内容形态 / 切角（切角是输入不是输出） |

## 常见错误（Common Mistakes）

- **把每个热点都凑成选题** —— 违「判断优先于凑数」。没桥就丢，宁可零选题。
- **靠共同名词硬拗桥** —— 只共享一个词就当接得住，跳过「同一批人 / 同一根神经」两探针。牵强，丢。
- **顺手把切角 / 内容方向 / 内容形态也产了** —— 越到下游三环节。切角只读不产，输出里这三个字段一个都不能有。
- **新造意图集外的意图**（如「提升声量 / 促进转化」）—— 孤儿选题，下游会被砍。指认不出意图集里哪条即弃。
- **凭记忆写热点原文** —— 跳过回读闭环，等于埋幻觉。必须按 record_id 逐字符回读。
- **改写「采纳状态」** —— 采纳是使用者的人工闸门，skill 只写「待复核」，绝不代填「采纳 / 弃用」。
- **把某个品类的红线写死在流程里** —— 红线一律运行时按 [品类 / 品牌] 标签加载，正文不内嵌任何具体品类的红线条目。
- **上下文文档缺失还硬跑** —— 缺链接即停报「缺项目选题上下文，需先蒸馏」，不猜路径、不自蒸馏。

## Red Flags

看到自己出现以下念头或动作，停下重判：
- 「这条勉强也能扯上」「顺嘴提一句就行」→ 多半是牵强，回去过四探针。
- 「今天没几条能出，先凑够数」→ 违判断优先于凑数，宁少勿滥。
- 「顺便把切角 / 方向 / 形态也定了吧」→ 越环节，立刻收回，只交选题。
- 「这条热点大概是这么说的」→ 没回读，别写，先按锚逐字符比对。
- 「这个红线应该没问题吧」→ 不确定就命中升级条件，停下交人裁。
